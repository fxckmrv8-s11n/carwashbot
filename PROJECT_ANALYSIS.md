# PROJECT_ANALYSIS.md — Этап 1: Анализ проекта

## 1. Что это за проект

CarWash Cloud — бот+веб-система для сети автомоек (кассы, машины, услуги,
сотрудники, финансы, отчёты). Backend: FastAPI/Flask-подобный `webapp/server.py`
(реальная точка входа, см. `Procfile → run_all.py → webapp.server:app`).
Есть Telegram-бот (`bot.py`), который открывает Mini App через `WEBAPP_URL`.

## 2. Два существующих интерфейса

### A. «Сайт» (webapp/static/*.html, кроме index.html)
Классический многостраничный сайт с общим сайдбаром:
`site-login.html → dashboard.html`, плюс `cars.html, cash.html, finance.html,
loyalty.html, reports.html, settings.html, branches.html, history.html,
workers.html`.
- Общий каркас/навигация — `site-common.js` (объект `NAV`).
- Тема — `site-theme.css`: тёмная, в стиле Revolut/премиальный SaaS
  (`--bg:#08090C`, фиолетово-синий акцент `--accent:#7C6CF6 → #4E9BFF`,
  зелёный/красный/оранжевый статус-цвета, `border-radius` 11–14px,
  мягкие тени, `backdrop-filter`, sidebar с активным пунктом и
  градиентной полосой слева).
- Авторизация — логин/пароль (`x-site-token`, `auth_web.py`), НЕ Telegram.
- Размер страниц: 155–692 строки каждая, разделены по смыслу (по одной
  странице на раздел).

### B. «Mini App» (webapp/static/index.html, 2194 строки)
Единый SPA-файл, единственный, кто подключает `telegram-web-app.js` и
использует `Telegram.WebApp` (haptics, showPopup/showConfirm/showAlert,
setHeaderColor, initData-авторизация через Telegram).
Открывается из бота как WebApp-кнопка.

Текущая навигация — 5 вкладок верхним/нижним таб-баром:
`Главная · Отчёты · Мойщики · Моя смена (скрыта, если не сотрудник) · Настройки`.

Внутри «Главной» уже реализовано много всего через switch-рендеры в JS
(не отдельные HTML-страницы, а функции, переключающие DOM):
`renderHome` (касса/машины дня), `renderAdd` (добавление/редактирование
машины со услугами), `renderReports`, `renderDayReport`, `renderAllCars`,
`renderHistory`, `renderWorkers`, `renderMyStats`, `renderEmployees`,
`renderScheduleWeek`, `renderSettings`.

Внутри `renderHome` уже встроены: касса, список машин, свайп-карточки
(`renderHomeSwipeCard`), расходы/доходы, разбивка оплаты по способам,
закрытие смены/архив.

## 3. Что уже зафиксировано в прошлой сессии (MINIAPP_AUDIT.md)

В проекте уже лежит `MINIAPP_AUDIT.md` от предыдущей работы над этим же
репозиторием. Ключевой вывод оттуда, который экономит нам время:
функционал сайта почти целиком уже перенесён в `index.html` (единственные
2 найденных на тот момент пробела — лояльность/скидки и
employee-stats — уже закрыты, судя по логу изменений в файле).

**Вывод: это не задача «перенести недостающие функции» — это задача
дизайна/архитектуры.** Функционал у mini-app почти при паритете с сайтом,
но **UX и визуально mini-app сильно проще сайта**:
- сайт: отдельные страницы, сайдбар, простор, карточки, премиальный тёмный
  UI;
- mini-app: один файл, 5 плоских вкладок, ощущается как утилита, а не как
  приложение уровня сайта.

## 4. Разрывы Mini App vs Сайт (визуально/структурно, не функционально)

| Аспект | Сайт | Mini App сейчас |
|---|---|---|
| Навигация | Sidebar с группами разделов | 5 вкладок в ряд, без группировки |
| Кол-во «страниц» | 10 отдельных HTML-страниц | 1 файл, switch между render-функциями |
| Тема | `site-theme.css`: продуманная палитра, радиусы, тени, градиенты | Своя, более простая тёмная тема внутри index.html (не переиспользует site-theme.css) |
| Разделы | Cars, Cash, Finance, Loyalty, Reports, Settings, Branches, History, Workers — каждый как полноценная страница с фильтрами/поиском | Всё свалено под «Главная» + отдельные вкладки только для Отчётов/Мойщиков/Настроек |
| Пустые состояния / загрузки | есть заготовки в site-common.js | частично, не единообразно |
| Финансы как раздел | отдельная страница `finance.html` | доходы/расходы — часть renderHome, отдельного раздела «Финансы» нет |
| Филиалы (branches) | отдельная страница управления филиалами | в mini-app нет отдельного раздела |

## 5. Технические ограничения, которые нужно сохранить

- Не трогаем backend API (`webapp/server.py`, `auth_web.py`, `history_log.py`,
  `notify.py`, `presets.py`, `xlsx_generator.py`, `employee_stats.py`) —
  только читаем существующие эндпоинты, ничего не переименовываем.
- Mini App обязана оставаться на `Telegram.WebApp` API (initData-авторизация,
  haptics, showPopup/showAlert/showConfirm, setHeaderColor) — это то, что
  отличает её от сайта и должно остаться.
- Точка входа не меняется: `bot.py` продолжает открывать тот же URL/файл.

## 6. Замечание про формулировку задачи

В задании упомянут «отдельно у меня есть полноценный сайт этого проекта» —
на деле сайт уже присутствует в этом же архиве (`webapp/static/*.html`,
кроме index.html) и я использовал его как референс дизайна и источник
недостающих разделов. Отдельный архив/репозиторий сайта не потребовался.
