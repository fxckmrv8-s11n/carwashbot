# Mini App Audit — прогресс

## ВАЖНО — реальная архитектура (уточнено в этой сессии, не пересканировать заново)

Это НЕ "одни и те же страницы под двумя способами входа". Это ДВА разных
интерфейса:

1. **«Сайт»** — классический многостраничный сайт:
   `site-login.html` → `dashboard.html` → (`cars.html`, `cash.html`,
   `finance.html`, `loyalty.html`, `reports.html`, `settings.html`,
   `branches.html`, `history.html`, `workers.html`). Навигация между ними —
   через общий сайдбар `NAV`, зашитый в `site-common.js` (строка ~137).
   Вход — `x-site-token` (логин/пароль, см. `auth_web.py`).
   Ни одна из этих страниц НЕ подключает `telegram-web-app.js`.

2. **«Mini App»** — `index.html` (2150 строк) — ОТДЕЛЬНОЕ single-page
   приложение. Единственный файл, который подключает
   `https://telegram.org/js/telegram-web-app.js` и использует Telegram.WebApp
   API. Открывается из бота через `WEBAPP_URL` (`bot.py:200-211`).
   Судя по размеру, `index.html` уже реализует часть функций сайта
   (кассу, машины, отчёты) внутри себя как SPA — нужно построчно сверить,
   каких разделов сайта там ещё НЕТ.

**Задача пользователя** = перенести функции, которых в `index.html` ещё нет,
в это SPA, сохранив Telegram Mini App UX (WebApp API), а не плодить ещё
несколько отдельных .html-страниц с SDK.

## Стадия 0 — инвентаризация (ЗАВЕРШЕНА)

Результат: `index.html` (mini-app SPA) уже покрывает почти всё —
дашборд/касса (`renderHome`), добавление и редактирование машин (`renderAdd`),
список машин (`renderAllCars`), расходы и доходы (finance.html дублирует то,
что уже есть в index.html через `/api/expense`, `/api/income` — отдельно
переносить не нужно), отчёты день/неделя/месяц/PDF/XLSX (`renderReports`),
история действий (`renderHistory`), сотрудники и график (`renderWorkers`,
`renderScheduleWeek`), настройки/доступы (`renderSettings`), личная
статистика (`renderMyStats`).

**Найдено ровно 2 реальных пробела:**

1. **Лояльность/скидки (`loyalty.html`) — ПОЛНОСТЬЮ отсутствует в mini-app.**
   Ноль упоминаний "loyalty" в index.html. API уже есть и используется
   сайтом: `POST /api/loyalty`, `DELETE /api/loyalty/{branch}/{idx}`
   (тело: `{branch, car_num, discount}`).
2. **Stage 12 (объединённая статистика по ролям) не подключён в mini-app.**
   `renderMyStats` и `renderWorkers` в index.html используют старые
   `/api/my-stats` / `/api/workers`, а не новые `/api/employee-stats` /
   `/api/employees-stats`. На сайте (`workers.html`) уже переключено —
   в mini-app ещё нет.

Всё остальное — уже сделано, разбивать на отдельные "страницы" не нужно.

## Ставка администратора + пустой отчёт (новая задача)

Запрошено: возможность закрыть/показать "пустой" отчёт (0 машин), если
сотруднику проставлена фикс-ставка (по умолчанию 1000₽), и такая же
кнопка "Ставка" — для администратора (раньше была только у мойщика).

**Найден и исправлен более серьёзный баг:** до этой правки, если за
смену не было ни одной машины, день вообще НЕ архивировался при начале
нового дня (`session.get("cars")` было единственным условием) — то есть
проставленная ставка мойщику/админу в пустой день молча ТЕРЯЛАСЬ бы при
`/newday`. Плюс `save_to_archive()` даже не сохраняла поля `fixed_rates`
и `admin_fixed_rate` в архив — то есть терялась ставка и в обычный
(непустой) день тоже при архивации. Оба бага исправлены.

### Изменения
- `sessions.py`: новый хелпер `session_has_data(session)` — "есть ли что
  сохранять" (машины/товары/расходы/доходы/ставка мойщика/ставка
  администратора). `save_to_archive()` теперь сохраняет `fixed_rates` и
  `admin_fixed_rate`.
- `calculator.py`: `admin_fixed_rate` — фикс-ставка администратора,
  прибавляется к `admin_salary` сверху (аналогично ставке мойщика),
  попадает в `role_earnings["администратор"]`.
- `webapp/server.py`: новые эндпоинты `POST/DELETE /api/admin-fixed-rate`
  (аналог `/api/fixed-rate`, но без имени — привязано к дежурному
  админу дня, `session["admin_name"]`).
- Все места, где раньше было `if session.get("cars"):` /
  `if not session["cars"] and not session.get("products")` для решения
  "есть ли данные за день" — заменены на `session_has_data()`:
  `handlers/admin.py` (старт нового дня — САМОЕ важное, иначе ставка в
  пустой день терялась безвозвратно), `handlers/reports.py`
  (`/report`), `handlers/cash.py` (`/summary`), `webapp/server.py`
  (`/api/newday`, `/api/reports/allreport`), `employee_stats.py`
  (агрегация текущей открытой смены).
- `webapp/static/workers.html` (сайт): у дежурного администратора
  появилась кнопка "Ставка" (иконка 💵), рядом с "Дежурство" — работает
  так же, как у мойщика: ввод суммы (по умолчанию предлагается 1000₽),
  0 — убрать ставку.
- В mini-app (`index.html`) кнопки "Ставка" НЕ добавлял — там её не было
  вообще ни для одной роли (это чисто функция сайта/бота на управление
  кассой администратором), только для просмотра. Если нужно — отдельным
  шагом.

### Проверено
- `session_has_data()` на пустом дне со ставкой → `True`; на полностью
  пустом → `False`.
- `calculate_summary()` на дне без единой машины с `fixed_rates` и
  `admin_fixed_rate` → корректно считает и мойщика, и администратора,
  `role_earnings` заполняется верно.
- Синтаксис всех изменённых `.py` и JS в `workers.html` — проверен.

## Команда /fix — задним числом проставить ставку за прошлый день

Запрошено: `/fix 14.07.2026` — добавить ставку 1000₽ Салиму, Саркису,
Роману, Артуру за уже закрытый (архивный) день.

В коде уже был прецедент — разовый скрипт `fix_100726_command`
(перезаписывал конкретный день полностью). Вместо ещё одного разового
скрипта сделал ОБЩУЮ команду `/fix`, которой можно чинить любой день:

`/fix ДД.ММ.ГГГГ Имя-Сумма Имя-Сумма ...`

Примеры:
- `/fix 14.07.2026 Салим-1000 Саркис-1000 Роман-1000 Артур-1000`
- `/fix 14.07.2026 admin-1000` — ставка дежурному администратору того дня
- `/fix 14.07.2026 Салим-0` — убрать ставку

Работает и по архивному дню, и по текущей ещё не закрытой смене (если
`session["date"] == дата`). Доступ — только владелец/админ филиала.

### Изменения
- `sessions.py`: `patch_fixed_rates(day, updates, admin_amount)` — общая
  логика правки ставок внутри словаря дня; `patch_archive_fixed_rates()`
  — применяет её к уже сохранённому архивному дню (возвращает False,
  если дня нет в архиве).
- `handlers/admin.py`: новая команда `fix_day_rates_command` (`/fix`).
- `bot.py`: команда зарегистрирована.

### Проверено
Изолированный тест (свой `DATA_DIR`, без затрагивания реальных данных):
патч архивного дня 14.07.2026 без единой машины → `fixed_rates`
записались, `calculate_summary()` посчитал зарплату каждому из четырёх
корректно (1000₽), `role_earnings` заполнен. Патч несуществующего дня
→ `False` (без исключения, безопасно). Синтаксис `sessions.py`,
`handlers/admin.py`, `bot.py` — проверен.

## /fix — уточнение: теперь СОЗДАЁТ пустой отчёт, если дня вообще не было

Раньше `/fix` мог только править УЖЕ существующий архивный день (возвращал
ошибку, если за эту дату вообще ничего не заводили). По просьбе — изменено:
если дня нет в архиве, `/fix ДД.ММ.ГГГГ ...` теперь сам создаёт пустой
отчёт (0 машин, 0 касса) за эту дату и сразу проставляет туда ставки.
Бот в ответе явно пишет "НОВЫЙ пустой отчёт... (создан этой командой)",
чтобы было видно, что день не существовал раньше.

`sessions.patch_archive_fixed_rates()` получил `create_if_missing` и
`admin_name` — при создании дня использует `admin_percent` по умолчанию.

Проверено изолированным тестом с нуля (архив полностью пустой) —
`/fix 14.07.2026 Салим-1000 Саркис-1000 Роман-1000 Артур-1000` создаёт
день и корректно считает зарплату всем четверым.
