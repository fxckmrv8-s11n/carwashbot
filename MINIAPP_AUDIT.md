# Mini App Audit — прогресс

## ВАЖНО — реальная архитектура (уточнено в этой сессии, не пересканировать заново)

Это НЕ "одни и те же страницы под двумя способами входа". Это ДВА разных
интерфейса:

1. **«Сайт»** — классический многостраничный сайт:
   `site-login.html` → `dashboard.html` → (`cars.html`, `cash.html`,
   `finance.html`, `loyalty.html`, `reports.html`, `settings.html`,
   `branches.html`, `history.html`, `workers.html`). Навигация между ними —
   через общий сайдбар `NAV`, зашитый в `site-common.js` (строка ~137).
   Вход — `x-site-token` (логин/пароль, см. `auth_web.py`).
   Ни одна из этих страниц НЕ подключает `telegram-web-app.js`.

2. **«Mini App»** — `index.html` (2150 строк) — ОТДЕЛЬНОЕ single-page
   приложение. Единственный файл, который подключает
   `https://telegram.org/js/telegram-web-app.js` и использует Telegram.WebApp
   API. Открывается из бота через `WEBAPP_URL` (`bot.py:200-211`).
   Судя по размеру, `index.html` уже реализует часть функций сайта
   (кассу, машины, отчёты) внутри себя как SPA — нужно построчно сверить,
   каких разделов сайта там ещё НЕТ.

**Задача пользователя** = перенести функции, которых в `index.html` ещё нет,
в это SPA, сохранив Telegram Mini App UX (WebApp API), а не плодить ещё
несколько отдельных .html-страниц с SDK.

## Стадия 0 — инвентаризация (ЗАВЕРШЕНА)

Результат: `index.html` (mini-app SPA) уже покрывает почти всё —
дашборд/касса (`renderHome`), добавление и редактирование машин (`renderAdd`),
список машин (`renderAllCars`), расходы и доходы (finance.html дублирует то,
что уже есть в index.html через `/api/expense`, `/api/income` — отдельно
переносить не нужно), отчёты день/неделя/месяц/PDF/XLSX (`renderReports`),
история действий (`renderHistory`), сотрудники и график (`renderWorkers`,
`renderScheduleWeek`), настройки/доступы (`renderSettings`), личная
статистика (`renderMyStats`).

**Найдено ровно 2 реальных пробела:**

1. **Лояльность/скидки (`loyalty.html`) — ПОЛНОСТЬЮ отсутствует в mini-app.**
   Ноль упоминаний "loyalty" в index.html. API уже есть и используется
   сайтом: `POST /api/loyalty`, `DELETE /api/loyalty/{branch}/{idx}`
   (тело: `{branch, car_num, discount}`).
2. **Stage 12 (объединённая статистика по ролям) не подключён в mini-app.**
   `renderMyStats` и `renderWorkers` в index.html используют старые
   `/api/my-stats` / `/api/workers`, а не новые `/api/employee-stats` /
   `/api/employees-stats`. На сайте (`workers.html`) уже переключено —
   в mini-app ещё нет.

Всё остальное — уже сделано, разбивать на отдельные "страницы" не нужно.

## Известные факты (чтобы не пересканировать)

- Точка входа в мини-апп: `bot.py` открывает `WEBAPP_URL` через кнопку меню
  (WebApp), см. `bot.py:200-211`.
- Реальный backend — `webapp/server.py` (не корневой `server.py` — тот
  устаревшая копия, НЕ используется, см. `Procfile` → `run_all.py` →
  `webapp.server:app`).
- Stage 12 (роли сотрудников) готов на бэкенде: `employee_stats.py`,
  `/api/employee-stats`, `/api/employees-stats`. Во фронтенде подключено
  пока только в `workers.html` (страница САЙТА, не mini-app).
- Общий сайдбар/навигация сайта — `site-common.js`, объект `NAV` (~строка 137).
